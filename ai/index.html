Getting Started with pgvector in YugabyteDB

This guide covers the complete lifecycle of working with AI vectors in YugabyteDB, from enabling the extension to tuning HNSW indexes for production workloads.

What is pgvector?

pgvector is a PostgreSQL extension that adds a native vector data type and distance operators (<->, <=>, <#>) to the database. YugabyteDB ships pgvector built-in from version 2.20, meaning you get distributed, fault-tolerant vector storage and HNSW indexing across multiple nodes — no separate vector database required.

1. Extension & Table DDL

CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  name TEXT,
  category TEXT,
  embedding vector(3) -- 3 dimensions for this example
);

2. Distance vs. Similarity

Databases optimize for Distance (smaller is better) rather than Similarity (larger is better). YugabyteDB maps similarity metrics into distances using specific operator classes.

🔬 Reading the Vector Data View

Open the Distance Metrics scenario in the Explorer and click each step button (L2, Cosine, Inner Product). The Vector Data View table at the bottom of the screen updates live — the column header switches between L2 Dist, Cos Dist, and IP Score, the rows re-sort by that metric, and the top-K nearest neighbors are highlighted green.

The same three labeled vectors — "Perfect Match", "Near", "Far" (L2 view), or "Parallel", "Orthogonal", "Opposite" (Cosine view) — produce completely different rankings under each metric. That single observation is the core intuition for metric selection.

Metric Operator Similarity Range Distance Mapping Use Case
L2 (Euclidean) <-> N/A 0 (identical) to ∞ Spatial data, unnormalized absolute distances.
Cosine <=> 1 (identical) to -1 1 - Similarity
Range: 0 to 2		 Text embeddings (OpenAI, etc) where angle matters.
Inner Product <#> ∞ (identical) to -∞ -1 * Similarity
Range: -∞ to ∞		 Fastest for embeddings pre-normalized to length 1.

Use Cases for Distance Metrics

Different embedding models and applications require different distance calculations. Here is a breakdown of when to use each.

1. L2 Euclidean Distance (<->)
Concept: Measures the physical straight-line distance between two points.
Use Case: Spatial data, RGB color matching, facial recognition, or anytime the absolute magnitude of the data is meaningful.
Example: Searching for the closest color to True Red [255, 0, 0]. A vector [250, 10, 5] is very close (low L2 distance), whereas [128, 0, 0] is much darker and further away in space.
2. Cosine Distance (<=>)
Concept: Measures the angle between two vectors, completely ignoring their length.
Use Case: Text embeddings and NLP (OpenAI, HuggingFace).
Example: Comparing the query "How to cook pasta" to a massive 10-page recipe document about "Boiling noodles". The document vector is huge (large magnitude) compared to the short query, but because they point in the same semantic "direction" (angle), Cosine considers them highly similar.
3. Inner Product (<#>)
Concept: Multiplies the vectors together. It takes both angle and magnitude into account.
Use Case: High-performance recommendation systems (like Netflix/Amazon predicting user click-through rates) OR replacing Cosine search for maximum speed.
Example: If you normalize all your text embeddings to a length of exactly 1 before inserting them into YugabyteDB, Inner Product mathematically produces the exact same ranking as Cosine, but requires significantly fewer CPU cycles to calculate.

3. Querying: Similarity vs. Distance

When querying, you often want to return the actual mathematical Similarity (e.g., "95% match") to the application, but you must use the Distance operator in the ORDER BY clause to utilize the HNSW index.

L2 (Euclidean) Example

-- L2 only has Distance, not a standard "Similarity" bound.
SELECT
  name,
  (embedding <-> '[0.12, -0.55, 0.89]') AS distance
FROM products
ORDER BY embedding <-> '[0.12, -0.55, 0.89]' ASC
LIMIT 5;

Cosine Example

-- 1 - Distance = Cosine Similarity
SELECT
  name,
  1 - (embedding <=> '[0.12, -0.55, 0.89]') AS similarity,
  (embedding <=> '[0.12, -0.55, 0.89]') AS distance
FROM products
ORDER BY embedding <=> '[0.12, -0.55, 0.89]' ASC -- Index used here
LIMIT 5;

Inner Product Example

-- -1 * Distance = Inner Product Similarity
SELECT
  name,
  -1 * (embedding <#> '[0.12, -0.55, 0.89]') AS similarity,
  (embedding <#> '[0.12, -0.55, 0.89]') AS distance
FROM products
ORDER BY embedding <#> '[0.12, -0.55, 0.89]' ASC
LIMIT 5;

4. Vector Normalization

Normalization scales a vector so that its total length (magnitude) is exactly 1, mapping it onto a "Unit Hypersphere".

The Formula

To normalize a vector V, divide every dimension by its Euclidean magnitude |V|.

-- 1. Calculate Magnitude: 
|V| = sqrt(v1² + v2² + ... + vn²)

-- 2. Normalize Vector:
V_norm = [v1 / |V|, v2 / |V|, ..., vn / |V|]

-- In YugabyteDB, use the built-in function:
SELECT l2_normalize('[3.0, 4.0]'::vector); -- Returns [0.6, 0.8]

Impact on Distance Metrics

When vectors are normalized (length = 1), the relationships between metrics mathematically collapse:

  • Cosine vs Inner Product: Cosine Similarity and Inner Product become exactly equal. Because Cosine calculates (A·B) / (|A||B|), if |A|=1 and |B|=1, the formula simply becomes A·B (Inner Product).
  • Performance: Inner Product requires fewer arithmetic operations than Cosine. Therefore, the most efficient architecture is to normalize your vectors in the application layer, store them in YugabyteDB, build an Inner Product index, and search using <#>.
  • L2 Distance: The relative ordering (ranking) of L2 distance becomes identical to Cosine and Inner Product when vectors are normalized.

Examples: Without vs With Normalization

-- V1: [3, 4]  (Magnitude 5)
-- V2: [6, 8]  (Magnitude 10, parallel to V1)

-- 🔴 WITHOUT NORMALIZATION:
SELECT '[3.0, 4.0]'::vector <-> '[6.0, 8.0]'::vector; -- L2 Distance: 5
SELECT '[3.0, 4.0]'::vector <=> '[6.0, 8.0]'::vector; -- Cosine Distance: 0 (Identical angle)
SELECT '[3.0, 4.0]'::vector <#> '[6.0, 8.0]'::vector; -- Inner Product Dist: -50

-- 🟢 WITH NORMALIZATION (Both become [0.6, 0.8]):
SELECT l2_normalize('[3.0, 4.0]') <-> l2_normalize('[6.0, 8.0]'); -- L2 Distance: 0
SELECT l2_normalize('[3.0, 4.0]') <=> l2_normalize('[6.0, 8.0]'); -- Cosine Distance: 0
SELECT l2_normalize('[3.0, 4.0]') <#> l2_normalize('[6.0, 8.0]'); -- Inner Product Dist: -1

5. HNSW Indexing & Execution Plans

Indexes are specific to the distance operator. You must create an index with the exact operator class you intend to search with.

Index Creation

-- Index for Cosine Distance (<=>)
CREATE INDEX idx_cosine ON products USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);

-- Index for L2 Distance (<->)
CREATE INDEX idx_l2 ON products USING hnsw (embedding vector_l2_ops);

-- Index for Inner Product (<#>)
CREATE INDEX idx_ip ON products USING hnsw (embedding vector_ip_ops);

Search Queries & EXPLAIN Plan

When you run a similarity search with an ORDER BY ... LIMIT clause, YugabyteDB will use the HNSW index to push down the KNN (K-Nearest Neighbors) scan.

-- Find the 5 most similar products based on Cosine Distance
EXPLAIN (ANALYZE, COSTS OFF)
SELECT name, embedding <=> '[0.12, -0.55, 0.89]' AS dist
FROM products
ORDER BY embedding <=> '[0.12, -0.55, 0.89]'
LIMIT 5;

Execution Plan Output:

 Limit (actual time=2.134..2.145 rows=5 loops=1)
   ->  Index Scan using idx_cosine on products (actual time=2.133..2.143 rows=5 loops=1)
         Order By: (embedding <=> '[0.12,-0.55,0.89]'::vector)
 Planning Time: 0.128 ms
 Execution Time: 2.181 ms

Notice the Index Scan indicating that the HNSW index successfully served the nearest neighbors without scanning the full 10,000 rows.

Tuning HNSW: The Highway Analogy

Think of an HNSW index as building a multi-layered highway network connecting thousands of cities (vectors) across a country. When you query the database, you are starting at a random drop-in point and trying to drive to your target destination as fast as possible.

Parameter Default The Analogy Pros & Cons
m
(Build Time)		 16 Highway Connections: The maximum number of roads connecting one city to another. Higher (e.g. 32, 64): More roads mean fewer dead-ends and shortcuts are easier to find (higher Recall for high-dimension data). Cons: Costs more concrete (RAM/Storage) and takes longer to build.

Optimal: 16 is fine for < 500 dims. Use 32+ for 1536+ dimension models (like OpenAI).
ef_construction
(Build Time)		 64 City Planning Effort: How exhaustively the city planners search for the absolute best neighboring cities to build roads to when a new city is added. Higher (e.g. 128, 200): Creates a near-perfect highway network, significantly boosting search accuracy. Cons: Makes `INSERT` and `CREATE INDEX` much slower.

Optimal: Set to at least 2x your `m` value (e.g., 128 or 200).
ef_search
(Run Time)		 40 GPS Route Candidates: While driving, how many alternate routes your GPS actively tracks and evaluates simultaneously. Higher (e.g. 100): You check more routes, virtually guaranteeing you find the shortest path (Recall approaches 100%). Cons: Checking 100 routes per hop costs CPU time, increasing query latency.

Optimal: Tune this per-session based on your latency SLA (e.g., 100). It MUST be >= your `LIMIT`. 
-- Tune search accuracy vs latency per session
SET hnsw.ef_search = 100;

SELECT name FROM products ORDER BY embedding <=> '[0.1, 0.2, ...]' LIMIT 5;

6. Building a RAG Pipeline

Retrieval-Augmented Generation (RAG) is the architecture used to ground Large Language Models (LLMs) with your private, proprietary data stored in YugabyteDB.

Step 1: Embed and Store Documents

First, you must process your knowledge base. Text is chunked into smaller paragraphs, sent to an embedding model (like OpenAI), and the resulting text and vector are stored together.

CREATE TABLE knowledge_base (id SERIAL, content TEXT, embedding vector(1536));
CREATE INDEX ON knowledge_base USING hnsw (embedding vector_cosine_ops);

-- App inserts chunked text alongside the vector generated by the LLM
INSERT INTO knowledge_base (content, embedding) VALUES ('YugabyteDB uses Raft...', '[0.12, -0.55, 0.89]');
Step 2: Semantic Retrieval

When a user asks a question (e.g., "How does replication work?"), the app converts the question into a vector and searches the database for the closest semantic matches.

-- Retrieve the top 3 most relevant document chunks to the user's question
SELECT content FROM knowledge_base
ORDER BY embedding <=> '[-0.01, 0.55, 0.89]' LIMIT 3;
Step 3: Augment and Generate

The application takes the text content returned by YugabyteDB and injects it directly into the prompt sent to the LLM. "Answer the user's question using ONLY the following context: [Context 1] [Context 2]...". This prevents hallucinations.

7. Hybrid Search

Pure similarity search is rarely enough. Applications usually need to combine Semantic Search (vectors) with Structural Search (relational data like tenant IDs, dates, and categories).

The Partial Index Approach

The most performant way to execute Hybrid Search in YugabyteDB is to build Partial HNSW Indexes on your most highly-filtered columns (like tenant_id or categorical partitions). This ensures the HNSW graph is perfectly pre-filtered.

-- Build an HNSW graph ONLY for electronics
CREATE INDEX idx_elec_cosine ON products USING hnsw (embedding vector_cosine_ops)
WHERE category = 'Electronics';

-- Build an HNSW graph ONLY for fruits
CREATE INDEX idx_fruit_cosine ON products USING hnsw (embedding vector_cosine_ops)
WHERE category = 'Fruits';

Hybrid Search Query

When you include the exact WHERE clause used in your partial index alongside the vector ORDER BY, YugabyteDB efficiently routes the query to the correct subgraph.

-- Find the most similar products THAT ARE ALSO strictly Electronics under $50
SELECT
  name,
  price,
  (embedding <=> '[0.1, 0.5, 0.8]') AS distance
FROM products
WHERE category = 'Electronics' -- Hits the partial index
  AND price < 50.00            -- Standard relational filtering
ORDER BY embedding <=> '[0.1, 0.5, 0.8]'
LIMIT 5;

Without a partial index, the database would have to do a "Post-Filter" (finding KNN results first, then throwing them away if they aren't Electronics) which hurts recall and performance.

Distributed Vector Search

When your data grows beyond a single node, YugabyteDB automatically shards the HNSW index across tablets. At query time, a gateway node scatters the search to every tablet leader in parallel, collects each shard's local Top-K results, and merges them into a single globally-sorted response — all transparently.

8. Production Checklist

Before deploying vector search to production, ensure you have addressed these critical items.

Choose the Right Dimension Size
Smaller dimensions (768) are faster but less expressive. Larger (1536, 3072) capture more nuance but consume more RAM and slow index builds. Match dimensions to your embedding model: OpenAI text-embedding-3-small = 1536, Cohere embed-v3 = 1024.
Connection Pooling for Vector Workloads
Vector search is CPU-intensive. Use connection pooling (PgBouncer, Odyssey) with transaction mode. Note that session-level settings like SET hnsw.ef_search must be re-applied per transaction when using transaction pooling.
Monitoring Index Usage
Use pg_stat_user_indexes to verify your HNSW index is being hit. If idx_scan = 0, your queries may be falling back to sequential scan — check that you're using the correct operator and have a LIMIT clause.
ef_search Must Be ≥ LIMIT
ef_search controls the candidate pool at query time. If it is smaller than your LIMIT K, the search cannot return K distinct results and recall silently degrades. Rule: ef_search ≥ LIMIT, and typically set to 2–5× LIMIT for high recall.
SET hnsw.ef_search = 100; -- LIMIT 20 → ef_search should be ≥ 20, ideally 60–100
-- Monitor HNSW index usage
SELECT indexrelname, idx_scan, idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
WHERE indexrelname LIKE '%hnsw%' OR indexrelname LIKE '%vector%';

-- Check index size
SELECT pg_size_pretty(pg_relation_size('idx_cosine')) AS index_size;

-- Rebuild HNSW index online
REINDEX INDEX CONCURRENTLY idx_cosine;

9. MCP Integration

What is MCP?

The Model Context Protocol (MCP) is an open standard that enables AI assistants (like Claude) to securely connect to external data sources and tools. Think of it as a universal plug-and-play adapter for LLMs. Instead of building custom API layers for every AI agent, MCP provides a standardized way to expose your enterprise data directly to the models.

What does it do for YugabyteDB?

By using the YugabyteDB MCP Server, you instantly grant AI assistants the ability to:

  • Introspect Schemas: The AI can automatically discover tables, columns, and relationships within your distributed database.
  • Query Live Data: The AI can write and execute SQL queries in real-time to answer user questions based on the latest data.
  • Perform Vector Search: The AI can natively perform similarity searches using pgvector without needing an intermediate middleware layer.

Example Queries for AI Assistants

Once integrated, you can ask your AI assistant to execute queries like the following to verify the setup and explore YugabyteDB-specific capabilities:

-- 1. Verify YugabyteDB Cluster Architecture
SELECT host, port, region, zone FROM yb_servers();

-- 2. Check Table Distribution & Sharding (Tablets)
SELECT table_name, num_tablets FROM yb_table_properties WHERE table_name = 'products';

-- 3. Run a Hybrid Vector Search
SELECT name, price FROM products
WHERE category = 'Electronics'
ORDER BY embedding <=> '[0.12, -0.55, 0.89]'
LIMIT 5;

By using MCP, you empower your AI to act as a highly capable database administrator and data analyst directly over your distributed YugabyteDB cluster.

10. Sampling Strategies

In LLM generation, the model outputs a probability distribution for the next token. Sampling is the process of selecting the actual token to print. Choosing the right strategy is critical for balancing "creativity" with "factuality".

In a YugabyteDB RAG pipeline, your vector search returns the Top-K candidate chunks (LIMIT K). These candidates enter the LLM's context window and influence the token probability distribution. Raising ef_search in YugabyteDB improves retrieval recall; the sampling parameters below then control how the LLM generates from that retrieved context.

A. Real-World Example: "The cat sat on the..."

Imagine the model predicts the following probabilities for the next token:

Token Probability Cumulative P Sampling Result
mat0.450.45Top-K (1), Top-P (0.5)
floor0.300.75Top-K (2), Out of P
bed0.150.90Top-K (3), Out of P
couch0.080.98Top-K (4), Out of P
moon0.021.00Top-K (5), Out of P

B. SQL Implementations

1. Simple Top-K (K=10)
SELECT token, probability FROM llm_output ORDER BY probability DESC LIMIT 10;
2. Top-P / Nucleus (P=0.70)
WITH ranked AS (
  SELECT token, probability,
         SUM(probability) OVER (ORDER BY probability DESC) as cum_p
  FROM llm_output
)
SELECT token FROM ranked WHERE cum_p <= 0.70;
3. Advanced: Temperature (T=0.7)
WITH temp_adjusted AS (
  SELECT token, EXP(LN(probability) / 0.7) as adjusted_p FROM llm_output
),
normalized AS (
  SELECT token, adjusted_p / SUM(adjusted_p) OVER() as final_p FROM temp_adjusted
)
SELECT * FROM normalized ORDER BY final_p DESC LIMIT 10;

C. Comparison & Synergy Table

Strategy Pros Cons
Greedy (K=1)Fastest, most deterministicRepetitive, lacks "soul"
Top-K (50)Reliable noise filterIgnores actual distribution shape
Top-P (0.70)Dynamic and nuancedComputationally heavier (Sort + Sum)
HybridBest of both worldsRequires careful tuning of K and P

D. Edge Case Analysis

  • K=0: Usually treated as "infinity" (disables Top-K).
  • P=1.0: Disables Top-P (includes all tokens in the tail).
  • High T + Low P: A "chaotic" setting that can lead to creative but hallucinated results.

11. Common Pitfalls

These are the most frequent mistakes developers encounter when working with pgvector in YugabyteDB.

❌ Mismatched Operator Class
Creating an index with vector_cosine_ops but querying with <-> (L2). The planner will NOT use the index and silently fall back to a sequential scan.
✅ Fix: Match the operator to the index. vector_cosine_ops → <=>, vector_l2_ops → <->, vector_ip_ops → <#>
❌ Forgetting the LIMIT Clause
HNSW indexes require an ORDER BY <distance> LIMIT K pattern. Without LIMIT, PG computes distances to all rows, completely defeating the index.
✅ Fix: Always include LIMIT. Use EXPLAIN to verify "Index Scan" appears in the plan.
❌ Using Inner Product Without Normalizing
Inner Product distance is only equivalent to Cosine Distance when all vectors have a magnitude of exactly 1. If you skip normalization, rankings will be wildly incorrect.
✅ Fix: Normalize in your app layer before INSERT, or use l2_normalize() in SQL. Then safely use <#> for maximum throughput.
❌ ef_search Too Low for Production
The default ef_search = 40 is fine for dev, but production workloads on high-dimensional data (1536+) often need 100-200 for acceptable recall (>95%).
✅ Fix: Benchmark with SET hnsw.ef_search = 100; and measure recall vs. latency. Increase until recall stabilizes.
❌ Mixing Embedding Models
Inserting vectors from text-embedding-ada-002 and querying with text-embedding-3-small. Different models produce incompatible vector spaces — results will be meaningless.
✅ Fix: Use the same model for both ingestion and query. Store the model name as a column for traceability.
